# ADR-016: Alembic Migration Safety — The include_object Firewall

**Status:** Accepted  
**Date:** 2026-02-22

---

## Context

The database schema will evolve as new user-facing features are added (e.g., preferences, route sharing, favourites). A migration framework is essential for safe, versioned schema changes.

**Flask-Migrate** (which wraps Alembic) is the standard tool for this. Its `--autogenerate` feature compares SQLAlchemy model metadata against the actual database schema to produce migration scripts automatically.

### The Trap

Although `user_db` is a separate database from `scenic_tiles` ([ADR-012](ADR-012-dual-database-segregation.md)), there are still risks:

1. **Future schema evolution** — If a developer accidentally points Alembic at the wrong database, it could detect tables it doesn't own (e.g., PostGIS `spatial_ref_sys`, `geometry_columns`) and generate `DROP TABLE` operations
2. **Reflected tables** — SQLAlchemy can discover tables via reflection that aren't part of our model metadata; Alembic interprets these as "unmanaged" and schedules them for removal
3. **Extension artefacts** — PostGIS creates system tables (`spatial_ref_sys`, `county`, `state`, etc. under the `tiger` schema) that could pollute autogenerated migrations

### Scale of Damage

If an unprotected `flask db migrate && flask db upgrade` runs:

```
INFO  [alembic.autogenerate.compare] Detected removed table 'spatial_ref_sys'
INFO  [alembic.autogenerate.compare] Detected removed table 'geometry_columns'
# → Drops PostGIS infrastructure → ALL spatial queries break
```

---

## Decision

**Install an `include_object` hook in the Alembic `env.py` that whitelists only tables declared in our SQLAlchemy metadata, rejecting everything else.**

### Implementation

When `flask db init` is run, the generated `migrations/env.py` must be modified:

```python
def include_object(object, name, type_, reflected, compare_to):
    """
    Firewall: Only allow Alembic to manage tables explicitly
    declared in our ORM models. All reflected/discovered tables
    from PostGIS extensions or other schemas are ignored.
    """
    if type_ == "table":
        # Only include tables that exist in our model metadata
        return name in target_metadata.tables
    # Allow columns, indexes, etc. on managed tables
    return True
```

This hook is then passed to `context.configure()`:

```python
context.configure(
    connection=connection,
    target_metadata=target_metadata,
    include_object=include_object,  # ← Firewall
    ...
)
```

### What It Protects Against

| Scenario | Without Hook | With Hook |
|----------|-------------|-----------|
| PostGIS `spatial_ref_sys` detected | `DROP TABLE spatial_ref_sys` | Ignored ✅ |
| `street_lighting` from osm2pgsql | `DROP TABLE street_lighting` | Ignored ✅ |
| New user model added | `CREATE TABLE users` | `CREATE TABLE users` ✅ |
| Column added to existing model | `ALTER TABLE ADD COLUMN` | `ALTER TABLE ADD COLUMN` ✅ |

---

## Consequences

### Positive

- **Catastrophe prevention** — Eliminates the risk of autogenerate destroying spatial infrastructure
- **Safe by default** — New developers running `flask db migrate` cannot accidentally damage the system
- **No false positives** — The whitelist approach means only explicitly declared models are managed

### Negative

- **Manual `env.py` edit** — The hook must be added after `flask db init`, which is a one-time manual step
- **New tables require model declaration** — Any table not in `db.Model` metadata will be invisible to migrations (this is the intended behaviour)

---

## Alternatives Considered

1. **Alembic `include_schemas` filtering** — Only introspect the `public` schema. Insufficient because `spatial_ref_sys` and `geometry_columns` live in `public`.

2. **Exclude list instead of include list** — Enumerate tables to skip (e.g., `exclude = {'spatial_ref_sys', 'geometry_columns'}`). Rejected because it requires maintenance as PostGIS adds new system tables, and forgetting an entry risks data loss.

3. **Separate Alembic config per database** — Run migrations only against `user_db` connection. This is already partially achieved by ADR-012 (separate database), but the `include_object` hook adds defense-in-depth in case of misconfiguration.

4. **Skip autogenerate entirely** — Write all migrations manually. Rejected because manual migrations are error-prone and slower to develop; autogenerate with proper filtering is the best of both worlds.

---

## Status: Deferred Implementation

The `include_object` hook is documented here for implementation when `flask db init` is first run during Docker deployment. The hook must be applied to `migrations/env.py` immediately after initialisation.
